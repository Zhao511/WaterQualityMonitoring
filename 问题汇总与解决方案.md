## 1. 通信层问题

### 1.1 LoRa 模块 Ping 无响应

| 项目 | 内容 |
|---|---|
| **提交** | `26a481d` |
| **严重程度** | 严重 — 导致 ESP32 网关无法确认 STM32 终端是否存活 |
| **现象** | ESP32 发送 ping 探测报文后，STM32 长时间无回复，表现为链路断开 |

**根因分析（2 个子问题）：**

1. **RSSI 查询破坏数据接收**：`IOT_Get_LoRa_RSSI()` 每 5 秒被 `vIOTTask` 调用一次，内部通过 `LoRa_SendATCmd("AT+RSSI?")` 查询信号。此操作将 LoRa 模块从**透传模式**切换到**AT 配置模式**，导致期间到达的 RF 数据（包括 ping 报文）被 AT 响应缓冲区吞掉，上层永远收不到。

2. **ISR 缓冲区溢出策略错误**：UART RX 中断中，当缓冲区满时原有代码执行 `*idx = 0` 清空全部缓冲区再写入新字节。当 ping 报文恰好跨缓冲区边界到达时，前半部分被清零，导致报文碎片化，无法被帧解析器识别。

**解决方案：**

- **RSSI 改为静态缓存**：在启动时查询一次 RSSI 并缓存，后续调用直接返回缓存值，不再频繁切换模块模式。
  ```c
  // Core/Src/iot_service.c
  static int8_t  s_rssi_cached = -55;
  static bool    s_rssi_queried = false;
  // 只在首次调用时查询，之后直接返回缓存
  ```
- **移除每 5 秒的 RSSI debug 打印**：`vIOTTask` 中不再每 5 个周期调用 `IOT_Get_LoRa_RSSI()` 打印日志。
- **ISR 缓冲区溢出改为丢弃新字节**：缓冲区满时不再清零，而是丢弃溢出字节，保护已积累的完整数据。
  ```c
  // Core/Src/stm32f10x_it.c
  if (*idx < buf_len)
      buf[(*idx)++] = data;
  // else: 缓冲区满，丢弃当前字节，保护已有数据
  ```

**经验教训：**
- 透传模式下的模块不应频繁切换到 AT 模式
- ISR 缓冲区管理应优先保护已接收数据，而非优先接收新数据
- 心跳检测不要与配置查询共享通信通道

### 1.2 LoRa 模块通信兼容性

| 项目 | 内容 |
|---|---|
| **提交** | `ef30016` |
| **严重程度** | 中等 — 影响不同固件版本 LoRa 模块的互通 |

**问题描述：**

本项目使用两种 LoRa 模块（ATK-LORA-01 和 E220 系列），它们使用不同的 AT 命令集和默认参数。在初始实现中：
- 两端未显式配置空中速率，依赖模块出厂默认值
- ESP32 在初始化时无条件执行出厂复位（`AT+RESET` 等），导致已正确配置的模块被反复重置
- 波特率扫描列表存在重复条目（115200 出现 2 次）

**解决方案：**

1. **两端显式配置空中速率 2.4kbps**：
   ```c
   // STM32 侧 Drivers/Src/lora.c
   LoRa_SendAT("AT+RATE=2\r\n", ...);   // ATK-LORA-01 命令
   // 失败则尝试 E220 兼容格式
   LoRa_SendAT("AT+PARAM=9,7,1,8\r\n", ...);  // SF9, 125kHz, 4/5, 8 前导
   ```
   ESP32 侧 `lora_uart.cpp` 同样增加速率配置。

2. **出厂复位改为条件触发**：仅当地址和信道配置**全部失败**时才执行复位，避免对已配好模块的无谓重置。

3. **波特率扫描列表去重**：`{115200, 115200}` → `{115200, 14400}`，覆盖更多可能速率。

**经验教训：**
- 异构 LoRa 模块通信必须显式设置所有关键参数
- 出厂复位是破坏性操作，应有条件而非无条件执行
- 波特率自适应扫描应覆盖模块可能的所有默认速率

### 1.3 LoRa Ping/Pong 链路探测机制

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 功能增强 |

**实现内容：**

ESP32 网关新增 LoRa 链路探测机制：
- 每 60 秒检查收到的 LoRa 帧数是否有变化
- 如帧数无变化（说明 STM32 可能离线），主动发送 `{"ping":1}` JSON 给 STM32
- STM32 收到后回复 `{"rsp":"ping", ...}` 
- 网关识别 `response:ping` 后**不上报云平台**（内部链路探测，避免云端收到无用数据）

```cpp
// esp32_gateway/src/main.cpp
static void lora_ping_check() {
    if (lora_frames_count() == frames_last) {
        lora_send("{\"ping\":1}");
    }
    frames_last = lora_frames_count();
}
```

## 2. 云平台对接问题

### 2.1 华为云不更新设备数据

| 项目 | 内容 |
|---|---|
| **提交** | `26a481d` |
| **严重程度** | 严重 — 云平台设备影子始终不更新，数据上报无效 |

**问题描述：**

设备上报属性到华为云 IoTDA 后，云端设备影子始终不更新。经排查，原因是属性上报 JSON 格式不符合华为云 IoTDA 规范。

**根因：**

`mqtt_report_property()` 发送的 JSON 缺少 `{"services": [...]}` 外层包装。

```
修复前: {"service_id":"Water_status","properties":{"pH":7.2,...}}
修复后: {"services":[{"service_id":"Water_status","properties":{"pH":7.2,...}}]}
```

**解决方案：**

在 `mqtt_report_property()` 中统一添加 `services` 数组外层：

```cpp
// esp32_gateway/src/mqtt_huawei.cpp
String wrapped = "{\"services\":[" + service_json + "]}";
g_mqtt.publish(g_topic_property.c_str(), wrapped.c_str());
```

**经验教训：**
- 华为云 IoTDA 属性上报格式为 `{"services":[{...}, ...]}`，支持批量上报
- 务必对照云平台官方文档验证 JSON 结构，而非依赖猜测
- 建议在开发阶段用 IoTDA 控制台的"设备调试"功能验证上报格式

### 2.2 LoRa Ping 响应误上报云平台

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 轻微 — 云端收到无意义数据 |

**问题描述：**

STM32 回复的 `ping` 响应通过 `response:ping` 命令转发给云平台，污染云端命令响应记录。

**解决方案：**

在 ESP32 `on_lora_receive()` 中增加过滤：

```cpp
if (cmd_name == "ping") {
    DEBUG_SERIAL.println("[FWD] LoRa ping pong (internal), skip cloud");
    return;
}
```

### 3.1 物模型字段补全

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 中等 |

```c
memcpy(alarm_out->device_id, IOT_DEVICE_ID_DEFAULT, IOT_DEVICE_ID_MAX - 1);
memcpy(alarm_out->rfid, rfid ? rfid : "", IOT_RFID_LEN - 1);
```

### 3.2 物模型多余字段清理

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 轻微 |

**问题描述：**

ESP32 `thing_model.h` 的 `WaterStatus` 结构体中包含 `turbidity`（浊度）字段，但该传感器在当前硬件版本中**未连接**（STM32 侧 `main.c` 固定将该值设为 0）。此冗余字段导致物模型与实际的 4 服务定义不一致。

## 4. 稳定性与健壮性问题

### 4.1 LoRa AT 命令鲁棒性

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 中等 |

**问题描述：**

不同 LoRa 模块（ATK-LORA-01 vs E220）使用不同的 AT 命令名实现相同功能（如 `AT+CH` vs `AT+CHANNEL` vs `AT+FREQ`）。初始代码仅使用单一命令格式，遇到不兼容模块时功能失效。

**解决方案：**

`set_lora_param` 命令实现采用**命令降级策略**：

```c
// Core/Src/iot_service.c
// 频率设置: 优先 AT+FREQ, 降级 AT+CH
snprintf(at_cmd, sizeof(at_cmd), "AT+FREQ=%d", freq);
if (LoRa_SendATCmd(at_cmd, ...) != 0) {
    snprintf(at_cmd, sizeof(at_cmd), "AT+CH=%d", freq);
    LoRa_SendATCmd(at_cmd, ...);
}
// 扩频因子: 优先 AT+SF, 降级 AT+RATE
// 功率: 优先 AT+POWER, 降级 AT+TXPOWER
```

同样，ESP32 的 `lora_init()` 中地址和信道配置也采用类似的多命令尝试策略。

**经验教训：**
- 面向不同固件/厂商的 AT 模块编程时，每个操作应提供 fallback 命令
- 定义统一的 `LoRa_SendATCmd()` 接口，封装命令兼容性逻辑

### 4.2 独立看门狗 (IWDG) 多任务心跳

| 项目 | 内容 |
|---|---|
| **提交** | 初始实现 `bcb5fe9` |
| **严重程度** | 系统级保障 |

**设计：**

STM32 实现多任务心跳 + 独立看门狗监控：

| 监控任务 | 角色 |
|---|---|
| `vWatchdogTask`（优先级 4） | 每秒检查所有 5 个应用任务的心跳标志；全部正常则喂狗，否则停止喂狗触发硬件复位 |
| 各应用任务 | 在每个循环周期设置自己的心跳标志 |

**关键参数：**
- IWDG 超时：约 4 秒（LSI 40kHz / 预分频 64 / 重装载 2499 = 4.0s）
- 检查周期：1 秒
- 看门狗任务优先级最高，确保即使低优先级任务饿死也能复位

## 5. 功能补全问题

### 5.1 告警管理功能（stub → 完整实现）

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 功能缺失 |

**问题描述：**

Alarm 服务的 3 条命令（`set_alarm_mode`、`clear_alarm`、`mute_alarm`）初始为 stub（空壳）实现，收到命令仅打印日志而不执行任何操作。

**解决方案：**

完整实现三条命令：

| 命令 | 实现 |
|---|---|
| `set_alarm_mode` | 设置 `g_alarm_mode` 为 `AUTO`（自动生成告警）或 `MANUAL`（暂停自动告警），`IOT_Alarm_Check()` 检查此标志 |
| `clear_alarm` | 将最近一条告警状态置为 `ALARM_STATUS_CLEARED`，支持按 `alarm_id` 精确清除或清除全部 |
| `mute_alarm` | 设置 `g_alarm_mute_until_tick` 静音截止时间戳，支持按告警类型过滤，到期自动恢复 |

告警检查逻辑增加前置判断：
```c
if (g_alarm_mode == ALARM_MODE_MANUAL) return 0;  // 手动模式跳过
if (xTaskGetTickCount() < g_alarm_mute_until_tick) return 0;  // 静音期内跳过
```

### 5.2 电子围栏 (GeoFence)

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 新增功能 |

**实现内容：**

- 基于 **Haversine 公式**计算球面距离
- `IOT_GeoFence_Check(lat, lon)` 检查当前位置是否在围栏半径内
- GPS 任务在每个定位周期调用围栏检查，越界时产生告警

```c
static float haversine_distance(float lat1, float lon1, float lat2, float lon2) {
    float dlat = to_rad(lat2 - lat1);
    float dlon = to_rad(lon2 - lon1);
    float a = sinf(dlat/2)*sinf(dlat/2) +
              cosf(to_rad(lat1))*cosf(to_rad(lat2))*
              sinf(dlon/2)*sinf(dlon/2);
    float c = 2.0f * atan2f(sqrtf(a), sqrtf(1.0f - a));
    return EARTH_RADIUS_M * c;
}
```


## 附录：关键经验教训

1. **透传与配置模式隔离**：LoRa 模块的透传模式和 AT 配置模式互斥，频繁切换会导致数据丢失。心跳/监控不应与参数配置共享通道。
2. **ISR 缓冲区管理**：缓冲区溢出时应优先保护已积累的完整数据，而非覆盖写入新字节。
3. **异构模块兼容**：面向不同厂商/固件的 AT 模块编程时，每个命令应提供 fallback，并显式设置所有关键参数。
4. **云平台协议对齐**：务必对照官方文档验证 JSON 格式（如华为云 `{"services":[...]}`），不能凭猜测。
5. **破坏性操作条件化**：出厂复位等不可逆操作应有条件触发，而非无条件执行。
6. **物模型严格一致**：云端和终端的所有服务、属性、字段必须严格一致，冗余字段会导致调试混乱。
