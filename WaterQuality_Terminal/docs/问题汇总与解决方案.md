## 1. 通信层问题

### 1.1 LoRa 模块 Ping 无响应

| 项目 | 内容 |
|---|---|
| **提交** | `26a481d` |
| **严重程度** | 严重 — 导致 ESP32 网关无法确认 STM32 终端是否存活 |
| **现象** | ESP32 发送 ping 探测报文后，STM32 长时间无回复，表现为链路断开 |

**根因分析（2 个子问题）：**

1. **RSSI 查询破坏数据接收**：`IOT_Get_LoRa_RSSI()` 每 5 秒被 `vIOTTask` 调用一次，内部通过 `LoRa_SendATCmd("AT+RSSI?")` 查询信号。此操作将 LoRa 模块从**透传模式**切换到**AT 配置模式**，导致期间到达的 RF 数据（包括 ping 报文）被 AT 响应缓冲区吞掉，上层永远收不到。

2. **ISR 缓冲区溢出策略错误**：UART RX 中断中，当缓冲区满时原有代码执行 `*idx = 0` 清空全部缓冲区再写入新字节。当 ping 报文恰好跨缓冲区边界到达时，前半部分被清零，导致报文碎片化，无法被帧解析器识别。

**解决方案：**

- **RSSI 改为静态缓存**：在启动时查询一次 RSSI 并缓存，后续调用直接返回缓存值，不再频繁切换模块模式。
  ```c
  // Core/Src/iot_service.c
  static int8_t  s_rssi_cached = -55;
  static bool    s_rssi_queried = false;
  // 只在首次调用时查询，之后直接返回缓存
  ```
- **移除每 5 秒的 RSSI debug 打印**：`vIOTTask` 中不再每 5 个周期调用 `IOT_Get_LoRa_RSSI()` 打印日志。
- **ISR 缓冲区溢出改为丢弃新字节**：缓冲区满时不再清零，而是丢弃溢出字节，保护已积累的完整数据。
  ```c
  // Core/Src/stm32f10x_it.c
  if (*idx < buf_len)
      buf[(*idx)++] = data;
  // else: 缓冲区满，丢弃当前字节，保护已有数据
  ```

**经验教训：**
- 透传模式下的模块不应频繁切换到 AT 模式
- ISR 缓冲区管理应优先保护已接收数据，而非优先接收新数据
- 心跳检测不要与配置查询共享通信通道

### 1.2 LoRa 模块通信兼容性

| 项目 | 内容 |
|---|---|
| **提交** | `ef30016` |
| **严重程度** | 中等 — 影响不同固件版本 LoRa 模块的互通 |

**问题描述：**

本项目**仅使用正点原子 ATK-LORA-01 模块**（E220 系列历史支持已于 `0abb855` 移除）。在初始实现中：
- 两端未显式配置空中速率，依赖模块出厂默认值
- ESP32 在初始化时无条件执行出厂复位（`AT+RESET` 等），导致已正确配置的模块被反复重置
- 波特率扫描列表存在重复条目（115200 出现 2 次）

**解决方案：**

1. **两端显式配置空中速率 2.4kbps**：
   ```c
   // STM32 侧 Drivers/Src/lora.c
   LoRa_SendAT("AT+RATE=2\r\n", ...);   // ATK-LORA-01 命令
   // E220 兼容格式 (已废弃, 代码已移除)
   // LoRa_SendAT("AT+PARAM=9,7,1,8\r\n", ...);  // SF9, 125kHz, 4/5, 8 前导
   ```
   ESP32 侧 `lora_uart.cpp` 同样增加速率配置。

2. **出厂复位改为条件触发**：仅当地址和信道配置**全部失败**时才执行复位，避免对已配好模块的无谓重置。

3. **波特率扫描列表去重**：`{115200, 115200}` → `{115200, 14400}`，覆盖更多可能速率。

**经验教训：**
- 异构 LoRa 模块通信必须显式设置所有关键参数
- 出厂复位是破坏性操作，应有条件而非无条件执行
- 波特率自适应扫描应覆盖模块可能的所有默认速率

### 1.3 LoRa Ping/Pong 链路探测机制

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 功能增强 |

**实现内容：**

ESP32 网关新增 LoRa 链路探测机制：
- 每 60 秒检查收到的 LoRa 帧数是否有变化
- 如帧数无变化（说明 STM32 可能离线），主动发送 `{"ping":1}` JSON 给 STM32
- STM32 收到后回复 `{"rsp":"ping", ...}` 
- 网关识别 `response:ping` 后**不上报云平台**（内部链路探测，避免云端收到无用数据）

```cpp
// esp32_gateway/src/main.cpp
static void lora_ping_check() {
    if (lora_frames_count() == frames_last) {
        lora_send("{\"ping\":1}");
    }
    frames_last = lora_frames_count();
}
```

### 1.4 LoRa UART 波特率匹配

| 项目 | 内容 |
|---|---|
| **提交** | `0abb855`, `9c25ed4` |
| **严重程度** | 严重 — 导致 ESP32 完全收不到 STM32 数据 |

**问题描述：**

ESP32 和 STM32 使用相同型号的 ATK-LORA-01 模块，但两端波特率反复调整仍无法通信。经过多轮测试发现：
- ESP32 模块 AT 配置模式固定 **115200**，9600 下返回乱码
- AT 写命令（`AT+BAUD`、`AT+ADDR=`、`AT+CH=`、`AT+RATE=`）全部返回 ERROR
- 模块只支持读查询命令（`AT`、`AT+ADDR?`）
- AT+BAUD 发送后模块会短暂切换波特率，但 `lora_set_baud()` 中 `LORA_SERIAL.end()` 产生的 UART BREAK 信号导致模块复位回 115200

**解决过程（4 次迭代）：**

| 轮次 | 尝试 | 结果 | 提交 |
|:---:|------|------|------|
| 1 | ESP32 统一 9600（与 STM32 一致） | AT 返回乱码 `�`，模块 AT 模式固定 115200 | — |
| 2 | ESP32 双波特率：AT=115200, 透传=9600 | AT+BAUD 发送后模块切到 9600，但 ESP32 UART 重启导致模块复位回 115200 | — |
| 3 | ESP32 固定 115200，STM32 改回 115200 | 两端统一 115200 | `0abb855` |
| 4 | **最终方案** | ESP32=115200, STM32=115200，全程不切换 | `9c25ed4` |

**解决方案：**

- ESP32 `LORA_BAUD_RATE = 115200`，`lora_init()` 全程 115200 不切换
- STM32 `LORA_BAUD_RATE = 115200`，两端统一
- 删除 `LORA_AT_BAUD_RATE`（不再有双波特率概念）

**经验教训：**
- 同一批次的 ATK-LORA-01 模块固件可能不同，有的支持 AT+BAUD，有的不支持
- `LORA_SERIAL.end()` + `begin()` 会产生 BREAK 信号，可能触发模块复位
- 波特率匹配问题应先用 USB-TTL 直连模块确认，而非通过代码猜测

### 1.5 ESP32 LoRa 模块 AT 命令兼容性

| 项目 | 内容 |
|---|---|
| **提交** | `9c25ed4` |
| **严重程度** | 中等 |

**发现：** ESP32 侧的 ATK-LORA-01 模块固件仅支持 2 条 AT 命令：

| 命令 | 支持 |
|------|:---:|
| `AT` | ✅ |
| `AT+ADDR?` | ✅ 返回 `+ADDR:00,00` |
| `AT+BAUD=` | ❌ |
| `AT+ADDR=` | ❌ |
| `AT+CH=` | ❌ |
| `AT+RATE=` | ❌ |
| `AT+RESET` | ❌ |
| 任何其他写命令 | ❌ |

模块的地址、信道、空中速率均依赖出厂默认值（address=0, channel=0, rate=2.4kbps），无法通过 AT 命令修改。

## 2. 云平台对接问题

### 2.1 华为云不更新设备数据

| 项目 | 内容 |
|---|---|
| **提交** | `26a481d` |
| **严重程度** | 严重 — 云平台设备影子始终不更新，数据上报无效 |

**问题描述：**

设备上报属性到华为云 IoTDA 后，云端设备影子始终不更新。经排查，原因是属性上报 JSON 格式不符合华为云 IoTDA 规范。

**根因：**

`mqtt_report_property()` 发送的 JSON 缺少 `{"services": [...]}` 外层包装。

```
修复前: {"service_id":"Water_status","properties":{"pH":7.2,...}}
修复后: {"services":[{"service_id":"Water_status","properties":{"pH":7.2,...}}]}
```

**解决方案：**

在 `mqtt_report_property()` 中统一添加 `services` 数组外层：

```cpp
// esp32_gateway/src/mqtt_huawei.cpp
String wrapped = "{\"services\":[" + service_json + "]}";
g_mqtt.publish(g_topic_property.c_str(), wrapped.c_str());
```

**经验教训：**
- 华为云 IoTDA 属性上报格式为 `{"services":[{...}, ...]}`，支持批量上报
- 务必对照云平台官方文档验证 JSON 结构，而非依赖猜测
- 建议在开发阶段用 IoTDA 控制台的"设备调试"功能验证上报格式

### 2.2 LoRa Ping 响应误上报云平台


| 项目 | 内容 |

## 4. 稳定性与健壮性问题

### 4.1 LoRa AT 命令鲁棒性

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 中等 |

**问题描述：**

不同 LoRa 模块（ATK-LORA-01 vs E220）使用不同的 AT 命令名实现相同功能（如 `AT+CH` vs `AT+CHANNEL` vs `AT+FREQ`）。初始代码仅使用单一命令格式，遇到不兼容模块时功能失效。

> **2026-06-16 更新**: E220 支持已完全移除，项目现仅使用 ATK-LORA-01 标准指令（`AT+CH`、`AT+RATE`、`AT+POWER`）。

**解决方案：**

`set_lora_param` 命令实现采用**命令降级策略**：

```c
// Core/Src/iot_service.c (历史实现, 现已简化为仅 ATK-LORA-01)
// 频率设置: AT+CH (ATK-LORA-01 标准)
snprintf(at_cmd, sizeof(at_cmd), "AT+CH=%d", freq);
LoRa_SendATCmd(at_cmd, ...);
// 扩频因子: AT+RATE (ATK-LORA-01 标准)
// 功率: AT+POWER (ATK-LORA-01 标准)
```

同样，ESP32 的 `lora_init()` 中地址和信道配置也采用类似的多命令尝试策略。

**经验教训：**
- 面向不同固件/厂商的 AT 模块编程时，每个操作应提供 fallback 命令
- 定义统一的 `LoRa_SendATCmd()` 接口，封装命令兼容性逻辑

### 4.2 独立看门狗 (IWDG) 多任务心跳

| 项目 | 内容 |
|---|---|
| **提交** | 初始实现 `bcb5fe9` |
| **严重程度** | 系统级保障 |

**设计：**

### 4.4 FreeRTOS 堆内存耗尽

| 项目 | 内容 |
|---|---|
| **提交** | `2cd02f8` |
| **严重程度** | 严重 — vIOTTask 增大栈后调度器无法启动 |

**问题描述：**

vIOTTask 栈从 512W 增大到 768W 后，系统无法启动（`[IoT] alive` 无输出）。

**根因：**

`configTOTAL_HEAP_SIZE = 8192` 字节不足以创建全部 6 个任务 + 5 个队列 + 3 个互斥量。

| 步骤 | 对象 | 消耗 (bytes) | 累计 | 剩余 |
|:---:|------|:---:|:---:|:---:|
| 1-8 | 5 队列 + 3 互斥量 | ~672 | 672 | 7520 |
| 9-12 | vWatchdog + vSensor + vGPS + vIOTTask(768W) | ~6232 | 6904 | 1288 |
| 13 | vRFIDTask (128W) + TCB | ~608 | 7512 | 680 |
| 14 | **vLEDTask (128W) + TCB** | **~608** | **8120** | **72 ❌** |

`xTaskCreate(vLEDTask)` 失败 → `LED_RGB_SetColor(RED) + for(;;)` 死循环 → 调度器从未启动。

**解决方案：**

1. `configTOTAL_HEAP_SIZE`: 8KB → **12KB**（STM32F103C8T6 共 20KB SRAM）
2. vIOTTask 栈: 768W → **512W**（`IOT_Process_Incoming` 峰值约 936B，2048B 足够）
3. vGPSTask 栈: 384W → **256W**

**经验教训：**
- 增大任务栈前必须计算总堆消耗
- STM32F103C8T6 的 20KB SRAM 中 FreeRTOS 堆建议 10-12KB

## 5. 功能补全问题

### 5.1 告警管理功能（stub → 完整实现）

```c
if (g_alarm_mode == ALARM_MODE_MANUAL) return 0;  // 手动模式跳过

```

### 5.2 电子围栏 (GeoFence)

| 项目 | 内容 |
|---|---|
| **提交** | `aa17b77` |
| **严重程度** | 新增功能 |

**实现内容：**

- 基于 **Haversine 公式**计算球面距离
- `IOT_GeoFence_Check(lat, lon)` 检查当前位置是否在围栏半径内
- GPS 任务在每个定位周期调用围栏检查，越界时产生告警

```c
static float haversine_distance(float lat1, float lon1, float lat2, float lon2) {
    float dlat = to_rad(lat2 - lat1);
    float dlon = to_rad(lon2 - lon1);
    float a = sinf(dlat/2)*sinf(dlat/2) +
              cosf(to_rad(lat1))*cosf(to_rad(lat2))*
              sinf(dlon/2)*sinf(dlon/2);
    float c = 2.0f * atan2f(sqrtf(a), sqrtf(1.0f - a));
    return EARTH_RADIUS_M * c;
}
```

### 5.6 LoRa_SendAT() AUX 时序 Bug

| 项目 | 内容 |
|---|---|
| **严重程度** | 严重 — 导致所有 AT 指令无响应 |

**问题描述：**

ATK-LORA-01 配置模式下 AUX 时序为：发送 `\r\n` 结束 → AUX 变 HIGH（忙）→ 模块处理 → 发响应 → AUX 变 LOW（空闲）。

原代码 `LoRa_SendAT()` 只等 AUX=LOW，跳过等 HIGH 步骤。发完指令后 AUX 可能还是 LOW（模块尚未开始处理），`LoRa_WaitAux` 瞬间返回，`LoRa_ReceiveData` 读到空数据。

**现象**：`AT` 返回 `WARN: AT no response`，后续指令响应串扰。

**解决方案**：发完指令后增加等 AUX=HIGH（200ms 超时），再等 AUX=LOW。同步修复 `LoRa_SendATCmd()`、`LoRa_SetAddress()`、`LoRa_SetChannel()`。

### 5.7 AT 指令集 V1.x → V3.0 迁移

| 项目 | 内容 |
|---|---|
| **严重程度** | 严重 — 所有 SET 指令静默失败 |

## 附录：关键经验教训

1. **透传与配置模式隔离**：LoRa 模块的透传模式和 AT 配置模式互斥，频繁切换会导致数据丢失。心跳/监控不应与参数配置共享通道。
2. **ISR 缓冲区管理**：缓冲区溢出时应优先保护已积累的完整数据，而非覆盖写入新字节。
3. **异构模块兼容**：面向不同厂商/固件的 AT 模块编程时，每个命令应提供 fallback，并显式设置所有关键参数。
4. **云平台协议对齐**：务必对照官方文档验证 JSON 格式（如华为云 `{"services":[...]}`），不能凭猜测。
5. **破坏性操作条件化**：出厂复位等不可逆操作应有条件触发，而非无条件执行。
6. **物模型严格一致**：云端和终端的所有服务、属性、字段必须严格一致，冗余字段会导致调试混乱。
7. **FreeRTOS 堆预算**：增大任务栈前必须计算总堆消耗，STM32F103C8T6 建议 heap 10-12KB。
8. **LoRa 模块 AT 命令兼容性先验证**：先用 USB-TTL 直连模块确认支持的 AT 命令集和固件版本（V1.x vs V3.0），再写代码。避免假设模块支持某命令。本次踩坑：代码用 V1.x 指令集（`AT+CH`/`AT+RATE`/`AT+POWER`），模块实际为 V3.0 固件（仅支持 `AT+WLRATE`/`AT+TPOWER`/`AT+UART`），所有 SET 指令静默失败数小时。
9. **UART 波特率切换慎用 `end()/begin()`**：`LORA_SERIAL.end()` 会产生 BREAK 信号，可能触发模块复位。
10. **配置模式 AUX 时序**：发送 AT 指令后必须等 AUX 先 HIGH（忙）再 LOW（空闲），仅等 LOW 会导致过早读取，响应为空。
11. **两端 RF 参数必须显式配置**：信道、速率、功率、地址必须在初始化时显式设置，不可依赖出厂默认。ESP32 端遗漏配置导致收到 10 万字节垃圾数据而 0 帧。

> **本版 (2026-06-16) 里程碑**：经过 V1.x→V3.0 AT 指令迁移 + AUX 时序修复 + 两端 RF 参数完整配置，**STM32 ↔ ESP32 LoRa 通信首次调通**。

### 8.3 帧协议：停等 ACK + 间隔控制 + 状态机接收

| 项目 | 内容 |
|---|---|
| **严重程度** | 架构重构 — 解决单向丢帧问题 |

**问题描述：** ESP32→STM32 单向不通（STM32 发 ESP32 收正常，反向丢帧）。原协议为纯 JSON + `\n` 分隔，无错误检测和重传机制。

**帧协议定义：**

```
数据帧: [0xAA] [LEN(1B)] [载荷(0~200B)] [XOR校验(1B)]
ACK帧:  [0xAA] [0x00] [0xAA]
NAK帧:  [0xAA] [0xFF] [0x55]
```

**STM32 接收状态机（`iot_transport.c`）：**
```
IDLE → GOT_HEADER → READING → GOT_CHECKSUM → 验证
                                                  ↓
                                  ┌─ 通过 → ACK → 解析载荷(A0/A1/JSON/ping)
                                  └─ 失败 → NAK → 丢弃复位
```

**ESP32 发送（`lora_uart.cpp` `lora_send_framed`）：**
1. 构造帧 → 等 AUX → TX
2. 等待 ACK（≤500ms）
3. 收到 NAK 或超时 → 重试（最多 2 次）
4. 帧间强制 ≥200ms 间隔

**联动改动：**
- ESP32 `main.cpp`：3 处 `lora_send()` → `lora_send_framed()`（A0/A1、JSON命令、ping）
- ESP32 `on_cloud_command()`：云端回复从"立即返回成功"改为"等 ACK 后才回复"

### 8.4 LoRa 接收数据丢失修复（2 处）

| 项目 | 内容 |
|---|---|
| **严重程度** | 严重 |

**问题 1 — ISR ORE 丢弃有效字节：** `USART3_IRQHandler` 中 ORE（溢出）处理仅读 DR 清标志，丢弃了 DR 中的有效字节。

**修复：** 将 `(void)USARTx->DR` 改为保存到缓冲区：
```c
uint8_t saved = (uint8_t)USARTx->DR;
if (*idx < buf_size) buf[(*idx)++] = saved;
```

**问题 2 — LoRa_ReceiveData 禁 ISR 时间过长：** 先禁 ISR，再 for 循环转移全部字节到环形缓冲。`lora_rx_index` 较大时禁 ISR 时间长，期间新到字节触发 ORE。

**修复：** 原子交换——仅禁 ISR 2 条指令（读+清零 index），数据转移在 ISR 开启下进行：
```c
__disable_irq();
count = lora_rx_index;
lora_rx_index = 0;
__enable_irq();
for (i = 0; i < count; i++) RingBuffer_Put(...);
```
